# 01 — Idea 模块化:按"扰动层"切分

LongLive 的 `longlive/methods/<idea>/` 接纳的 idea 数量会随时间增长(Bridge、OFT、未来的 NFT 变体、sparse-attn 等)。为避免每个 idea 重新发明 trainer / config / 注册逻辑,把 diffusion 训练 pipeline 拆成 **6 个扰动层**,每个 idea 显式声明自己触碰哪几层,只通过该层已有的 seam 注入。

## 抽象层级

| 层 | 内容 | 已有 seam | 当前位置 | 例子 idea |
|---|---|---|---|---|
| **L0 数据** | 配对/无配对/motion-ref 采样 | dataset 类 | `longlive/utils/dataset.py` | 数据挖掘、ref 选择策略 |
| **L1 参数化** | 可训练参数怎么挂到冻结主干 | adapter dispatch | `longlive/utils/lora_utils.py` + YAML `adapter.type` | **OFT** / DoRA / SVDiff |
| **L2 前向 hook** | 运行时改 attn / FFN 张量 | hook injector | `longlive/model/motion_hooks.py:MotionAttnInjector` | motion-attn / sparse-attn |
| **L3 教师 score** | 蒸馏的目标信号怎么算 | 子类 DMD | `longlive/model/dmd.py:_compute_kl_grad*` | motion-DMD / **Bridge** / NFT |
| **L4 损失形式** | 学生 vs 目标怎么比 | DenoisingLoss 插件 | `longlive/utils/loss.py:NAME_TO_CLASS` | flow / consistency / Bridge residual |
| **L5 训练循环** | 阶段顺序 / 何时切换什么 | 子类 trainer | `longlive/trainer/ScoreDistillationTrainer` | 多阶段调度(罕见) |

## 设计纪律

- **新 idea 不能新增层。** 要新增层意味着已有抽象 leaks,先讨论再写代码。
- **每个 idea 声明自己触碰哪几层。** 写在 `longlive/methods/<idea>/README.md` 第一段。设计审查 = 检查这个声明是否最小、是否准确。
- **idea 之间正交是好信号。** 两个 idea 投影到不同层 → 可同时启用,组合不需要互相知道对方存在。

## 已有 idea 的层投影

- **motion-DMD**(已落地):L2(`MotionAttnInjector` 注入 V_ref attn)+ L3(`DMD._compute_kl_grad_with_motion` 把 teacher score 改成 `(1-β)·v_real + β·v_real_motion`)。
- **Bridge**(规划中):L3(子类 `DMD` 把 `_compute_kl_grad` 改成 `teacher_score + Δb_φ(x_t, t)`,`Δb_φ` 不另开网络,由 LoRA 参数承载)+ 可选 L4(残差损失若不能用现有 `flow` / `dmd_switch` 表达)。trainer / adapter 不动。
- **OFT**(规划中):仅 L1。adapter dispatch 注册一个 `"oft"`;`longlive/utils/oft_utils.py` 提供正交参数化 + attach。DMD / loss / trainer 不动。

Bridge 与 OFT 完全正交 → 同一 YAML 可同时配 `adapter.type: oft` + `distribution_loss: bridge_dmd`。

## 注册方式:启动时自动扫描 + side-effect 注册

约定:

1. core 注册表(`NAME_TO_CLASS`、adapter dispatch、`distribution_loss` 分发)保持现状,不感知 idea 存在。
2. trainer 启动时一次性扫描 `longlive/methods/*/` 全部 import(`pkgutil.iter_modules` 或显式遍历),让每个 method 的 `__init__.py` 跑一遍。
3. 每个 `longlive/methods/<idea>/__init__.py` 用 side-effect 把自己注册到 core 表里。
4. YAML 选哪个 active 仍由 `adapter.type` / `distribution_loss` / 等具体字段决定;扫描只是把候选都加载进注册表。

效果:加一个 idea = 新建一个目录;无需动 YAML、core、trainer 任何配置或代码。

**`__init__.py` 纪律(强制)**:只允许 (a) `from .X import Y` 形式的相对 import、(b) `register_*(...)` 注册调用。**不允许任何 heavy module-level 代码** —— 不 instantiate model、不 `torch.compile`、不读盘、不分配显存。否则所有 run 启动时都为这个 idea 付代价,与"未启用 = 零开销"的承诺破坏。

**键冲突**:注册函数发现 key 已存在 → raise,不静默覆盖。

**性能影响**:训练时零开销(未 active 的 method 代码永远不调用)。启动时多 import 几个轻量 module 是 ms 级,跟 FSDP wrap + 14B teacher load 比可忽略。CPU 内存 +几十 MB 可忽略,GPU 零。

## `longlive/methods/<idea>/` 文件骨架

```
longlive/methods/bridge/
  __init__.py            # register BridgeDMD into longlive/model/dmd dispatch
  README.md              # 一段话:touches L3 (+L4),why this cut
  score.py               # class BridgeDMD(DMD): override _compute_kl_grad
  losses.py              # only if L4 needs new loss type
  configs/cat_dunk_v1.yaml  # 完整 copy + override `configs/longlive_train_motion.yaml`
longlive/methods/oft/
  __init__.py            # register adapter type "oft"
  adapter.py             # 正交参数化、attach
  configs/cat_dunk_oft_v1.yaml
```

L0 / L2 / L5 的文件只在真要用时再开,不预留空目录。

## YAML 写全,不做继承

每个 method config(如 `longlive/methods/bridge/configs/cat_dunk_v1.yaml`)是 `configs/longlive_train_motion.yaml` 的完整 copy + override,**不引入 `_base_:` 继承机制**。trainer 入口零特殊逻辑。代价:method config 跟 base config 漂移时要手动同步 —— 接受这个成本以换取 trainer / OmegaConf 路径的简单性。

## 决策点

- ✅ **注册机制**:启动时自动扫描 `longlive/methods/*/`(`__init__.py` 纪律守住,效率不是问题,见上一节)。
- ✅ **YAML 继承**:不做,每个 method config 写全。
- ⏳ **Bridge `Δb_φ` 的承载**:全由现有 LoRA 参数学(L3-only)/ 新增小型残差 head(L1+L3)。**待 Bridge 实现规划时单独讨论**。
